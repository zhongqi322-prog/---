# P6退款与异常处理

更新时间：2026-07-01

## 1. 目标

定义真实收款后的异常处理边界，避免出现已扣款但无报告、重复支付、支付成功但状态丢失、无法退款等问题。

## 2. 必须处理的异常

| 场景 | 系统处理 |
|---|---|
| 用户取消支付 | 订单置为 `cancelled` 或保持 `pending_payment` 直到超时 |
| 支付失败 | 订单置为 `payment_failed`，允许重新支付 |
| 支付超时 | 订单置为 `expired`，重新购买需新订单 |
| 重复点击支付 | 复用未过期订单或通过幂等键阻止重复创建 |
| 重复回调 | 只处理第一次有效成功回调 |
| 金额不一致 | 拒绝入账并标记人工核查 |
| 支付成功但报告生成失败 | 订单置为 `fulfillment_failed`，允许重试或退款 |
| 用户重复支付 | 保留一笔有效订单，其余进入退款或人工处理 |
| 用户申请退款 | 记录退款原因、状态和处理人 |

## 3. 退款状态

最小退款字段：

- `refund_id`
- `order_id`
- `amount`
- `reason`
- `status`
- `provider_refund_id`
- `requested_at`
- `processed_at`

退款状态：

- `refund_requested`
- `refund_processing`
- `refunded`
- `refund_failed`
- `refund_rejected`

## 4. 退款说明原则

付费前必须让用户知道：

- 本服务为传统文化参考和问题整理，不承诺现实结果。
- 报告一经生成，是否支持退款必须提前说明。
- 支付成功但系统未生成报告，应支持补发报告或退款。
- 重复支付应支持核对后退回重复款项。

## 5. 客服与运营处理

第一版至少需要：

- 订单号可复制。
- 用户可提交问题描述。
- 运营可按订单号查询支付状态、报告状态和退款状态。
- 人工处理必须留下备注。

## 6. 不得做

- 不用前端本地状态作为真实支付成功依据。
- 不隐藏失败订单。
- 不用恐吓、焦虑或“错过改运”等话术阻止退款。
- 不承诺通过追加付费改变现实结果。

## 7. 开发前验收

- 异常状态已进入订单状态机。
- 支付回调失败重试策略已确定。
- 报告生成失败后的重试和退款策略已确定。
- 退款说明已进入付费页或用户协议。
