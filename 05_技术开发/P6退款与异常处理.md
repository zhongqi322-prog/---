# P6退款与异常处理

更新时间：2026-07-01

## 1. 目标

定义真实收款后的异常处理边界，避免出现已扣款但无报告、重复支付、支付成功但状态丢失、无法退款等问题。

## 2. 必须处理的异常

| 场景 | 系统处理 |
|---|---|
| 用户取消支付 | 订单置为 `cancelled` 或保持 `pending_payment` 直到超时 |
| 支付失败 | 订单置为 `payment_failed`，允许重新支付 |
| 支付超时 | 订单置为 `expired`，重新购买需新订单 |
| 重复点击支付 | 复用未过期订单或通过幂等键阻止重复创建 |
| 重复回调 | 只处理第一次有效成功回调 |
| 金额不一致 | 拒绝入账并标记人工核查 |
| 支付成功但报告生成失败 | 订单置为 `fulfillment_failed`，允许重试或退款 |
| 用户重复支付 | 保留一笔有效订单，其余进入退款或人工处理 |
| 用户申请退款 | 记录退款原因、状态和处理人 |

## 3. 退款状态

最小退款字段：

- `refund_id`
- `order_id`
- `amount`
- `reason`
- `status`
- `provider_refund_id`
- `requested_at`
- `processed_at`

退款状态：

- `refund_requested`
- `refund_processing`
- `refunded`
- `refund_failed`
- `refund_rejected`

## 3.1 退款权限阶段策略

用户已确认：退款权限正式阶段再开通。

因此当前阶段规则为：

- 测试阶段不做真实收款，不产生真实退款。
- 测试阶段只验证订单状态、mock 退款状态和客服处理流程。
- 正式阶段必须先开通微信支付退款权限，再上线真实收款。
- 如果退款权限未开通，不得将支付按钮接入真实扣款。
- 正式上线前必须完成至少一次内部退款链路验证，确认平台退款号、退款状态通知和订单状态同步都可追踪。

## 4. 退款说明原则

付费前必须让用户知道：

- 本服务为传统文化参考和问题整理，不承诺现实结果。
- 报告一经生成，是否支持退款必须提前说明。
- 支付成功但系统未生成报告，应支持补发报告或退款。
- 重复支付应支持核对后退回重复款项。

## 5. 客服与运营处理

第一版至少需要：

- 订单号可复制。
- 用户可提交问题描述。
- 运营可按订单号查询支付状态、报告状态和退款状态。
- 人工处理必须留下备注。

## 5.1 客服 SLA

第一版建议：

| 场景 | 首次响应 | 处理目标 | 备注 |
|---|---:|---:|---|
| 支付成功但无报告 | 4 小时内 | 24 小时内补发或退款 | 优先级最高 |
| 重复支付 | 1 个工作日内 | 3 个工作日内完成核对 | 核对平台交易号 |
| 用户申请退款 | 1 个工作日内 | 3 个工作日内给出处理结果 | 以付费说明为准 |
| 报告删除请求 | 1 个工作日内 | 3 个工作日内完成 | 对账字段不删除 |
| 普通咨询 | 2 个工作日内 | 5 个工作日内完成 | 不承诺命理结果 |

## 5.2 人工权限

后台至少区分：

- 只读客服：只能查看订单状态、报告状态和用户提交的问题。
- 运营处理人：可标记补发、退款申请、删除申请和处理备注。
- 管理员：可执行退款确认、敏感字段删除和异常订单关闭。

权限要求：

- 所有人工操作必须记录操作者、时间、原因和前后状态。
- 客服不得查看完整支付账户信息。
- 客服不得修改报告内容来迎合用户现实结果。
- 管理员不得绕过订单状态直接发放付费报告。

## 5.3 对账节奏

第一版建议：

- 每日核对前一日支付成功订单与系统 `paid` 订单。
- 每日核对退款申请与平台退款状态。
- 每周抽查 `paid` 但未 `fulfilled` 的订单。
- 每月导出订单汇总用于财务核对。

必须人工核查：

- 金额不一致。
- 平台显示成功但系统无订单。
- 系统 `paid` 但平台未成功。
- 同一平台交易号对应多个订单。
- 同一用户短时间多笔重复支付。

## 6. 不得做

- 不用前端本地状态作为真实支付成功依据。
- 不隐藏失败订单。
- 不用恐吓、焦虑或“错过改运”等话术阻止退款。
- 不承诺通过追加付费改变现实结果。

## 7. 开发前验收

- 异常状态已进入订单状态机。
- 支付回调失败重试策略已确定。
- 报告生成失败后的重试和退款策略已确定。
- 退款说明已进入付费页或用户协议。
- 客服 SLA、人工权限和对账节奏已确认。
- 微信支付退款权限已在正式阶段开通并验证。
