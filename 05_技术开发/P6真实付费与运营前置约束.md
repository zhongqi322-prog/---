# P6真实付费与运营前置约束

## 1. 目的

P6 涉及真实收款、订单、报告保存、用户隐私和运营闭环，不能从 P5 的 mock 流程直接改成真实支付。

本文件用于约束 P6 开始前必须重新设计的内容。

## 2. 禁止直接复用 mock 支付

P5 的 mock 支付只用于原型验证。

P6 禁止：

- 把 mock 支付按钮直接接入真实支付。
- 只改按钮文案就上线收款。
- 没有订单号就收款。
- 没有退款、失败、取消、重复支付处理就上线。
- 没有用户同意和隐私说明就保存报告。

## 3. P6 必须先设计

进入真实开发前必须完成：

- 真实支付方案选择。
- 订单状态机。
- 支付成功 / 失败 / 取消 / 超时处理。
- 报告保存策略。
- 用户隐私和敏感信息边界。
- 风险提示和退款说明。
- 祈福心愿履约边界。
- 后台订单查询方式。
- 用户反馈和客服处理方式。

## 4. 订单状态最小集合

第一版至少包含：

- `created`
- `pending_payment`
- `paid`
- `payment_failed`
- `cancelled`
- `refunded`
- `fulfilled`

## 5. 报告保存边界

报告保存必须明确：

- 保存哪些字段。
- 保存多久。
- 用户是否可删除。
- 是否保存出生日期、出生时间、住址、手相图片等敏感信息。
- 是否需要脱敏。
- 是否需要导出或删除入口。

## 6. 隐私与合规底线

P6 不得承诺：

- 一定灵验。
- 保证实现。
- 改命。
- 消灾必成。
- 治病。
- 驱邪。
- 法术。
- 官方寺庙保证。

所有付费页面必须说明：

```text
本服务为传统文化参考、心愿记录与祝福表达，不承诺现实结果。
```

## 7. P6 开始前验收

P6 开发前必须先输出：

- 支付与订单设计文档。
- 报告保存与隐私边界文档。
- 退款与异常处理文档。
- 小范围用户测试反馈总结。
