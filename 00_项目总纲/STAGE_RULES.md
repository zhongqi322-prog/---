# STAGE_RULES.md｜阶段制 Codex 工作规则

## 1. 本文件目的

本项目采用“阶段制工作流”，避免项目运行时间变长后 Codex 上下文过大、质量下降。

用户只需要发出三类指令：

1. 开始下一阶段
2. 进入当前阶段审核
3. 审核通过，进入下一阶段

Codex 必须根据当前阶段、上一阶段交接文和本文件自动判断工作范围。

## 2. 阶段制核心原则

### 2.1 新阶段最小读取顺序

每个新阶段开始时，Codex 必须优先读取：

1. `00_项目总纲/PROJECT_STAGE.md`
2. `00_项目总纲/CONTEXT.md`
3. `08_阶段交接/上一阶段_交接文.md`
4. 当前阶段直接相关的 1-3 个文件

禁止无目的读取全项目。

### 2.2 阶段结束必须交接

每个阶段结束时，Codex 必须在 `08_阶段交接/` 创建：

`P阶段编号_阶段名称_交接文.md`

### 2.3 用户只做阶段决策

用户不需要手动判断当前应由哪个 Agent 执行。Codex 必须根据当前阶段自动选择主要 Agent。

## 3. 项目阶段划分

### P0 项目规则阶段

- 主要 Agent：总控架构 Agent
- 目标：建立目录、核心规则、CONTEXT、阶段制规则、古籍录入规范、紫微推理架构和发布路线。
- 完成标志：项目结构清晰，基础规则齐全，P1 可直接开始。

### P1 产品设计阶段

- 主要 Agent：产品设计 Agent、AI报告与提示词 Agent
- 目标：完成页面流程、功能入口、付费路径、报告模板、祈福规则和紫微付费报告设计。
- 完成标志：每个功能有流程文档，每个报告有模板结构，前端可直接开发。

### P2 可点击原型阶段

- 主要 Agent：前端开发 Agent
- 目标：创建 Next.js 网站，实现页面、表单、mock 报告、价格解锁、轻量记录和移动端适配。
- 完成标志：本地可运行，所有入口存在，只使用 mock 数据，不接真实 AI 或支付。

### P3 古籍知识库阶段

- 主要 Agent：古籍知识库 Agent
- 目标：建立首批结构化古籍片段、`data/classics/*.json`、术语表和引用规范。
- 完成标志：八字、紫微、风水、易经、解梦、相术均有带来源的 mock 古籍片段。

### P4 推理与报告阶段

- 主要 Agent：命理算法 Agent、紫微斗数 Agent、AI报告与提示词 Agent
- 目标：完成各模块 mock 推理，并统一接入古籍出处和风险提醒。
- 完成标志：用户填写资料后可生成无绝对化断语的结构化 mock 报告。

### P5 发布测试阶段

- 主要 Agent：发布测试 Agent、前端开发 Agent
- 目标：部署 Vercel，完成移动端、链接、mock 支付、风险话术与上线检查。
- 完成标志：网站公开可访问，可进行真实用户测试。

### P6 真实付费与运营阶段

- 主要 Agent：后端/支付相关 Agent、产品设计 Agent、发布测试 Agent
- 目标：接入真实支付、订单、报告保存、真实祈福履约、反馈和复购机制。
- 完成标志：可以真实收款、保存报告和跟踪订单。

## 4. PROJECT_STAGE.md 规则

Codex 必须维护 `00_项目总纲/PROJECT_STAGE.md`，至少包含：

- 当前阶段
- 当前状态
- 当前主要 Agent
- 当前阶段目标
- 当前阶段必须完成
- 上一阶段交接文
- 下一阶段

## 5. 阶段交接文模板

每个阶段结束时必须创建 `08_阶段交接/PX_阶段名称_交接文.md`，包含：

1. 本阶段目标
2. 本阶段完成内容
3. 新增文件
4. 修改文件
5. 当前有效决策
6. 下阶段必须读取的文件
7. 下阶段禁止或不建议读取的文件
8. 未完成事项
9. 风险提醒
10. 给下阶段 Agent 的建议
11. 用户需要确认的事项

## 6. 阶段审核规则

用户说“进入当前阶段审核”时，Codex 必须停止新增开发并创建：

`08_阶段交接/PX_阶段名称_审核报告.md`

审核报告必须检查：

- 本阶段目标与文件是否完成
- 是否存在范围偏离
- 是否加入英文版、宝宝起名或分享返佣
- 是否出现保证灵验、绝对化断语或无出处结论
- 必须修复项与建议优化项
- 是否建议进入下一阶段

## 7. 用户指令响应规则

### 开始下一阶段

1. 读取 `PROJECT_STAGE.md`
2. 读取 `CONTEXT.md`
3. 读取上一阶段交接文
4. 判断当前阶段并自动选择主要 Agent
5. 开始执行当前阶段任务

### 进入当前阶段审核

1. 停止新增开发
2. 执行审核并生成审核报告
3. 列出必须修复项
4. 等待用户确认

### 审核通过，进入下一阶段

1. 确认当前阶段审核报告存在
2. 生成或更新最终交接文
3. 更新 `PROJECT_STAGE.md`
4. 将状态改为下一阶段准备中
5. 等待用户说“开始下一阶段”

## 8. 上下文节约规则

每个阶段开始时不得读取全项目。优先读取：

1. `CONTEXT.md`
2. `PROJECT_STAGE.md`
3. 上一阶段交接文
4. 当前阶段相关文件

除非任务明确要求，否则不得读取全部古籍 raw 文件、全部开发日志、全部产品文档或全部代码文件。

## 9. 一句话总结

用户负责控制阶段，Codex 负责根据阶段交接文自动判断上下文、Agent、任务和输出。
