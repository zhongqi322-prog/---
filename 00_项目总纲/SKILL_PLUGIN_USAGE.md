# SKILL_PLUGIN_USAGE.md｜阶段 Skill 与插件使用规则

## 1. 目的

本文件规定本项目各阶段适合使用的 Skill 或插件。

每个阶段开始前，Codex 必须先自行评估当前任务需要使用哪些 Skill 或插件。凡命中适用场景的，必须先读取对应 Skill 说明或插件使用说明，并按其规则执行，再开始正式任务。

## 2. 强制执行规则

### 2.1 任务开始前必须评估

每次收到以下指令时，Codex 必须先执行 Skill / 插件评估：

1. 开始下一阶段
2. 进入当前阶段审核
3. 审核通过，进入下一阶段
4. 用户提出具体开发、测试、发布、古籍处理、文档处理、网页检查任务

### 2.2 命中即使用

如果任务明显匹配某个 Skill 或插件能力，必须使用。

使用方式：

1. 先读取对应 `SKILL.md` 或插件说明。
2. 按说明执行必要步骤。
3. 在执行前简要说明本次使用的 Skill / 插件和原因。
4. 如果工具不可用，必须记录原因，并使用次优方案。

### 2.3 不得滥用

不得为了形式而使用无关 Skill 或插件。

只使用与当前阶段、当前任务直接相关的 Skill 或插件。

## 3. 全阶段通用 Skill / 插件

### 3.1 karpathy-guidelines

适用：

- 多步骤软件开发
- Bug 修复
- 重构
- 代码审核
- 架构或实现取舍
- 构建、测试、验证链路

要求：

- 复杂开发任务开始前必须读取并遵循。
- 简单文档修改、单行修正、非编程资料整理不强制使用。

### 3.2 GitHub 插件

适用：

- GitHub 仓库状态检查
- 提交、推送、PR、Issue、CI 检查
- 远程仓库同步
- GitHub 作为唯一真实数据源的验证

要求：

- 涉及远程仓库、PR、CI、GitHub 页面时优先使用。
- 本地 `git` 命令仍可用于常规提交和状态检查。

### 3.3 Browser 插件

适用：

- 本地网页打开
- localhost 页面检查
- 点击测试
- 表单测试
- 弹窗测试
- 移动端页面视觉检查

要求：

- 前端页面变更后，若本地服务可运行，应优先使用 Browser 检查。
- Browser 不稳定或不可用时，使用构建、路由 HTTP 检查或 Playwright 作为替代，并记录原因。

### 3.4 webapp-testing Skill

适用：

- Web 应用端到端流程测试
- 表单、路由、弹窗、localStorage、移动端验证
- 发布前页面检查

要求：

- P2、P5 阶段审核或修复时优先使用。

### 3.5 deploy-pipeline Skill

适用：

- Vercel / 部署流程
- 构建产物检查
- 上线前检查
- 发布后 smoke test

要求：

- P5 发布测试阶段必须评估并优先使用。

## 4. 各阶段推荐 Skill / 插件

### P0 项目规则阶段

主要使用：

- GitHub 插件：远程仓库、同步状态、治理文件。
- karpathy-guidelines：涉及仓库结构或多文件规则修改时使用。

通常不需要：

- Browser
- webapp-testing
- deploy-pipeline

### P1 产品设计阶段

主要使用：

- karpathy-guidelines：当产品文档影响后续工程结构时使用。
- GitHub 插件：提交和远程同步时使用。

可选：

- documents：如果用户提供 Word 项目文档。
- pdf：如果用户提供 PDF 资料。
- spreadsheets：如果用户提供表格资料。

### P2 可点击原型阶段

必须评估并优先使用：

- karpathy-guidelines：前端开发前使用。
- Browser 插件：本地页面检查、点击检查。
- webapp-testing Skill：流程测试和审核。
- GitHub 插件：提交、推送、远程同步。

可选：

- playwright：Browser 插件不稳定或需要更强自动化测试时使用。

### P3 古籍知识库阶段

必须评估：

- pdf：用户提供古籍 PDF 时使用。
- documents：用户提供 Word / 文档格式古籍时使用。
- spreadsheets：用户提供 Excel / CSV / 表格化古籍时使用。
- GitHub 插件：结构化数据入库和同步。

必须遵守：

- 真实古籍内容由用户提供或用户确认来源。
- 不得编造古籍原文、书名、章节或出处。

### P4 推理与报告阶段

必须评估：

- karpathy-guidelines：算法、报告结构、数据接入开发时使用。
- openai-docs：接入 OpenAI API、结构化输出、提示词工程时使用。
- webapp-testing：报告生成流程测试时使用。
- GitHub 插件：提交、CI、远程同步时使用。

要求：

- 推理、验证、报告结构必须尽量写成确定性代码。
- 不得把合规边界完全交给模型自由发挥。

### P5 发布测试阶段

必须评估并优先使用：

- deploy-pipeline：部署和上线流程。
- Browser 插件：公开页面和移动端检查。
- webapp-testing：上线前流程测试。
- GitHub 插件：CI、提交、发布状态。
- github:gh-fix-ci：CI 失败时使用。

可选：

- Netlify 插件不作为默认选择；当前发布路线优先 Vercel。

### P6 真实付费与运营阶段

必须评估：

- karpathy-guidelines：真实支付、订单、报告保存属于高风险开发。
- GitHub 插件：PR、CI、发布控制。
- webapp-testing：支付前后流程测试。
- openai-docs：真实 AI 报告生成接入时使用。

可能需要：

- 支付平台官方文档或插件，后期根据实际支付方案确定。
- spreadsheets：用户反馈、订单导出、运营数据整理。

## 5. 明确暂不使用或谨慎使用

### Notion

当前不建议使用。

原因：本项目已确定 Obsidian / GitHub / Codex 三者统一，GitHub 是唯一真实数据源。

### Netlify

当前不作为默认发布插件。

原因：项目发布路线优先 Vercel。

### Figma

仅当用户明确要求正式设计稿、设计系统或页面视觉稿时使用。

### Twilio

当前无短信、电话、验证码需求，暂不使用。

### HyperFrames / Remotion

仅当用户要求宣传视频、讲解视频或动态内容时使用。

### imagegen

仅当用户要求生成图像、品牌图、页面插图或宣传素材时使用。

## 6. 主任务检查清单

每个主任务开始前，Codex 必须回答并执行：

1. 当前任务属于哪个阶段？
2. 当前主要 Agent 是谁？
3. 当前任务是否是编程开发？
4. 当前任务是否涉及网页测试？
5. 当前任务是否涉及 GitHub / CI / 发布？
6. 当前任务是否涉及古籍 PDF、Word、表格或结构化数据？
7. 当前任务是否涉及 OpenAI API 或真实 AI 接入？
8. 命中的 Skill / 插件有哪些？
9. 是否已读取对应 Skill / 插件说明？
10. 若未使用某个明显相关工具，原因是什么？

---

## 7. 一句话总结

阶段开始前先评估 Skill / 插件。

命中适用场景必须先使用，再开始任务。

不滥用无关工具，不跳过必要工具。
