# CONTEXT.md｜项目当前最小上下文

## 1. 项目定位

本项目是中文版“传统术数古籍 AI 解读平台”。

当前只做中文版 MVP，英文版暂停，等中文版盈利后再考虑。

项目核心不是让 AI 无依据算命，而是基于：

* 古籍出处
* 结构化规则
* 八字 / 紫微斗数 / 风水 / 易经 / 解梦 / 相术等传统体系
* AI 白话解释
* 风险边界提醒

为用户生成有出处、有解释、有边界的咨询报告。

---

## 2. 当前阶段目标

当前 P1 阶段目标是完成产品流程、功能入口、付费转化路径和报告模板设计。

可点击网站原型属于 P2 阶段，不属于当前 P1 阶段。

第一版要求：

* 使用 mock 数据
* 不接真实 AI
* 不接真实支付
* 不做完整账户系统
* 使用 localStorage 做轻量记录
* 最终发布到 Vercel
* 所有报告必须有古籍出处模块

---

## 3. 已确定决策

以下决策不再反复讨论，除非用户明确修改：

1. 只做中文版，英文版暂停。
2. 不使用 Claude Code 和 OpenClaw，只使用 Codex 多 Agent。
3. Obsidian / GitHub / Codex 必须是同一个项目系统。
4. GitHub 仓库是唯一真实数据源。
5. Obsidian 是项目大脑界面，不是第二套知识库。
6. Codex 是执行者，不能创建独立知识库。
7. 今日黄历放在首页，不作为独立主项目。
8. 不做宝宝起名，改为姻缘解惑。
9. 保留八字和紫微斗数。
10. 紫微斗数作为付费增强内容。
11. 保留祈福心愿、周公解梦、手相解析。
12. 暂不做分享返佣。
13. 第一版使用 mock 支付。
14. 第一版必须可发布到 Vercel。
15. 古籍内容由用户通过其他方式收集；进入 P3 古籍知识库阶段或具体需要真实古籍片段时，Codex 必须向用户索取，不得自行编造。

## 5. 祈福心愿模块边界

祈福心愿保留：

* 心愿祈福
* 点亮心愿灯
* 为家人祈愿
* 线下祈愿记录
* 照片 / 视频回传说明

但当前第一版只做 mock 流程和页面展示。

不得承诺：

* 一定灵验
* 保证实现
* 改命
* 消灾必成
* 治病
* 驱邪
* 法术
* 官方寺庙保证

必须加入说明：
本服务为传统文化、心愿记录与祝福表达，不承诺现实结果。

## 6. 古籍出处原则

所有解读报告必须包含：

* 古籍依据
* 书名
* 原文片段
* 白话解释
* 与用户问题的关系
* 风险提醒

禁止：

* 编造古籍原文
* 无出处输出结论
* 绝对化预测
* 恐吓用户

## 11. 上下文更新规则

当项目发生以下变化时，必须更新本文件：

1. 功能范围变化
2. 技术路线变化
3. Agent 分工变化
4. 付费策略变化
5. 古籍录入规则变化
6. 发布路线变化
7. 用户明确修改核心决策

更新时要求：

* 只保留当前有效决策
* 删除过期内容
* 不把开发日志写进 CONTEXT.md
* 不把详细任务过程写进 CONTEXT.md

---

## 12. CONTEXT.md 的长度规则

本文件必须保持简洁。

建议长度：

* 最好控制在 1500-3000 字
* 不要写详细开发历史
* 不要写完整古籍内容
* 不要写每个页面的全部细节

详细内容应放到对应模块文件中。

## 13. 一句话总结

Obsidian 是理解项目的大脑界面。
GitHub 是唯一真实数据源。
Codex 是执行这个项目的开发者。

三者必须是同一个项目，不允许分裂。
