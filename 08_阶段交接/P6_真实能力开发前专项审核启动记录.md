# P6 真实能力开发前专项审核启动记录

更新时间：2026-07-08

## 1. 阶段结论

小范围用户复测准入信号已满足，可以进入真实支付、报告保存、真实 AI 开发前的专项审核阶段。

本阶段不是直接开发真实支付，也不是直接接入真实大模型 API。只有专项审核通过后，才允许进入真实能力开发实施。

## 3. 本阶段目标

1. 确认微信支付真实收款前置条件。
2. 确认订单、退款、异常处理和客服处理边界。
3. 确认报告保存、删除、导出、隐私告知和数据留存策略。
4. 确认真实 AI API 的输入、输出、安全拦截、失败降级和成本控制。
5. 形成是否允许进入真实能力开发的审核结论。

## 4. 支付专项审核

进入真实支付开发前必须确认：

- 微信支付商户号是否已开通。
- 主体是否已升级为个体工商户或其他正式经营主体。
- 可用支付产品：H5、JSAPI、Native 扫码等具体类型。
- 结算账户是否已完成认证。
- 支付域名是否已在微信支付商户平台配置。
- 回调地址是否具备稳定 HTTPS 访问能力。
- 退款权限是否已开通。
- 是否能完成至少一次内部退款链路验证。
- 客服联系方式、退款处理负责人和处理时限是否明确。

未确认前禁止事项：

- 不得把“模拟解锁”按钮改成真实扣款按钮。
- 不得以前端本地状态作为支付成功依据。
- 不得上线真实扣款但无退款处理能力。
- 不得在退款权限未开通前对真实用户收费。

## 5. 订单与异常专项审核

真实支付开发前必须确认：

- 订单状态机是否采用 `created -> pending_payment -> paid -> fulfilling -> fulfilled` 主链路。
- 是否支持 `payment_failed`、`cancelled`、`expired`、`fulfillment_failed`、`refunded`。
- 同一订单重复回调是否幂等处理。
- 同一平台交易号是否只能绑定一个订单。
- 金额、币种、订单号、支付状态是否由服务端校验。
- 支付成功但报告生成失败时，是否支持重试、补发或退款。
- 重复支付、金额不一致、平台成功但系统无订单等异常是否进入人工核查。

## 6. 报告保存与隐私专项审核

真实报告保存前必须确认：

- 报告正文默认保存周期。
- 用户是否可以在“我的记录”删除报告正文。
- 删除报告后，订单和支付对账字段如何保留。
- 原始表单输入是否保存；如保存，保存哪些结构化摘要。
- 出生时间、出生地、联系方式、图片、住址等字段是否最小化处理。
- 是否提供隐私说明、用户同意机制和删除路径。
- 是否明确不保存支付密码、验证码、银行卡号、完整支付账户标识。

第一版建议：

- 报告正文默认保存 180 天。
- 图片类数据第一版不做真实上传保存。
- 地址类数据不保存精确门牌。
- 订单对账字段按支付、财务和售后需要保留，但不展示完整支付账户信息。

## 7. 真实 AI 专项审核

真实 AI API 接入前必须确认：

- 模型只负责语言组织和白话解释，不能自由编造古籍出处。
- 古籍引用只能来自已审核结构化数据。
- 风险提醒由代码固定注入或校验覆盖。
- 输出必须符合结构化 schema。
- 高风险输入和输出必须拦截或改写。
- 不向模型发送支付信息、身份证件、银行卡、验证码等无关敏感信息。
- 模型失败时是否有降级策略：返回测试摘要、提示稍后重试、人工补发或退款。
- 是否记录模板版本、模型版本、古籍片段 ID、风险规则版本和失败原因。
- 是否有成本上限、超时上限和重试上限。

禁止输出：

- 保证发财、保证复合、保证灵验。
- 医疗、投资、法律或重大人生决策替代建议。
- 改命、驱邪、治病、消灾必成等承诺。
- 诱导高额消费或连续加购。

## 9. 阶段通过标准

本阶段通过需同时满足：

- 用户确认支付资质和产品权限。
- 用户确认价格和退款处理原则。
- 用户确认报告保存、删除和隐私策略。
- 用户确认真实 AI 接入范围和安全边界。
- 技术方案明确后完成一次复审评分。

未通过前，仍不得进入真实支付、真实 AI 或完整账户系统开发。
